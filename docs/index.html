Crucible Matrix Testing Methodology

1. Test Architecture

Benchmarks are executed on a dual-GPU AMD system consisting of two 16GB RX 7800 XT cards (32GB total VRAM). Due to the lack of hardware interconnects like NVLink, cards communicate via a bifurcated PCIe Gen 4 x8/x8 topology on the motherboard, offering up to 16 GB/s bidirectional bandwidth per card.

2. Control vs. SOTA Edge

To accurately measure driver (ROCm/Vulkan) and engine compiler optimization over time, we maintain fixed control configurations (e.g., Llama 3 8B Instruct). The rest of the matrix features cutting-edge models (e.g., Gemma 4, Qwen 3.6, Llama 4) to test hardware memory limits.

3. Raw Speed Optimization

Our framework targets absolute maximum speed rather than system stability. This includes utilizing nightly builds compiled from source, aggressive quantization methods (FP8, AWQ, EXL2), speculative decoding pipelines, and custom ROCm kernels (AITER/PagedAttention).

Execution & Measurement Protocol

The automated benchmark executor runs periodically in an isolated state to prevent resource contention. Each test configuration undergoes the following strict pipeline:

  • Provisioning & Verification: Model weights are downloaded and verified. The runtime engines are either compiled from their latest nightly commits or fallback to the latest stable release. Exact commit hashes and driver metadata are recorded.
  • Warm-up: The engine processes 2 to 3 dummy requests to warm up the GPU memory allocation, load KV caches, and compile any runtime shaders.
  • Testing Loop: The target benchmark prompts are executed a minimum of 5 times ($N \ge 5$) with strict timeouts to prevent hard hangs. We record the median and $p95$ latencies for Time to First Token (TTFT) and Time Per Output Token (TPOT).
  • Teardown: Inference processes are aggressively terminated, and GPU VRAM is completely flushed before the next test begins.

⚠️ Blind Spots & Limitations

While this testing matrix identifies peak potential throughput and minimum latency, it does not capture long-term system stability or qualitative degradation. Nightly compiler builds and experimental quantization formats are prone to runtime memory leaks, numerical precision drift, and ROCm kernel panics. Additionally, PCIe Gen 4 x8 bandwidth bottlenecks are heavily masked under low batch sizes but show extreme degradation during parallel expert sharding (e.g., MoE Tensor Parallelism).
