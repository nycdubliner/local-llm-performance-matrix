Hardware Architecture & Constraints

Benchmarks are executed on a dual-GPU AMD system consisting of two 16GB Radeon RX 7800 XT graphics cards (providing 32GB total VRAM). Due to the lack of hardware interconnects (like NVLink or xGMI) on these consumer-grade cards, communication between GPUs operates over the host PCIe bus.

The motherboard topology features a bifurcated PCIe Gen 4 x8/x8 link. This setup limits theoretical maximum unidirectional throughput to 16 GB/s per card. Inter-GPU messaging and layer synchronizations are highly sensitive to bus collisions under parallel expert routing or tensor parallel operations.

The 6 Dimensions of Local LLM Testing

To capture a complete engineering landscape, our Crucible Matrix tests performance across six orthogonal vectors:

1. Inference Engines

Testing nightlies and source-compiled compilers: vLLM (highly concurrent request queues), llama.cpp (lightweight CPU/GPU memory split), and MLC LLM (Vulkan shader compiles).

2. Model Architectures

Evaluating dense transformer configurations alongside Mixture of Experts (MoE) to compare memory load and bandwidth bounds.

3. Quantization Formats

Measuring the latency-throughput trade-offs of low-precision representations, including GGUF, AWQ, EXL2, and native FP8.

4. Runtime Optimizations

Actively testing speculative decoding, multi-token prediction (MTP), paged KV caching, and customized flash kernels.

5. GPU Sharding Strategies

Investigating the overhead differences between Tensor Parallelism (TP) (inter-GPU tensor splitting) and Pipeline Parallelism (PP) (stage-by-stage sequential layer layout).

6. Workload Scenarios

Benchmarking single-request chat (Batch=1) for latency limits alongside batch-processing pipelines (Batch=16) for aggregate generation rates.

Execution & Measurement Protocol

The automated benchmark runner runs periodically in an isolated system state to eliminate CPU/GPU resource pollution. Each test case runs through the following strict protocol:

  1. Provisioning & Verification: Target weights are fetched and checksum-verified. Runtimes are rebuilt from source commits to utilize latest driver paths, recording driver/driver-library tags.
  2. System Warm-up: The engine executes 3 dummy requests to allocate KV cache, trigger ROCm shader compiler caching, and establish stable GPU clocks.
  3. Testing Loop: Standardized prompts are executed a minimum of 5 times ($N \ge 5$) with strict watchdog timeouts. We capture the median and $p95$ metrics for Time to First Token (TTFT) and Time Per Output Token (TPOT).
  4. Aggressive Teardown: Process groups are forcefully terminated (SIGKILL) and VRAM allocations are completely flushed using driver calls before starting the next test.

⚠️ Qualitative Limitations & Blind Spots

While this tracking matrix identifies peak potential throughput and minimum latency, it does not capture long-term system stability or qualitative degradation. Nightly compiler builds and experimental quantization formats are prone to runtime memory leaks, numerical precision drift, and ROCm kernel panics. Additionally, PCIe Gen 4 x8 bandwidth bottlenecks are heavily masked under low batch sizes but show extreme degradation during parallel expert sharding (e.g., MoE Tensor Parallelism).
